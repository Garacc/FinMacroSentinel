# 代码修改行为规范

## 核心原则

### 1. 先分析，后行动
- 遇到问题先分析根本原因
- 不要急着改代码，先理解现有代码的逻辑
- 确认问题真正所在再动手

### 2. 不盲目改版本
- 原代码使用的版本、依赖有既定原因
- 不随意修改 Node 版本、依赖版本等基础配置
- 如需修改，必须有明确的技术理由

### 3. 最小改动原则
- 只改必须改的地方
- 保持原有逻辑不变
- 避免画蛇添足

### 4. 验证优先
- 修改后先在本地验证
- 不确定的效果先截图或描述给用户确认
- 再推送到远端

### 5. 不确定就问
- 对于不明确的需求，先和用户确认
- 对于不确定的修改，先描述方案再执行

---

## 典型错误反思

| 错误做法 | 正确做法 |
|----------|----------|
| 用户说一个问题就改一个地方 | 先分析问题根源，确认后再改 |
| 随意改版本号 | 不改版本，除非有明确理由 |
| 加多余的代码 | 只改最小必要的地方 |
| 不验证就推送 | 先本地测试确认 |
| 急着给方案 | 先理解问题，再给方案 |
